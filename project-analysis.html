Project Thesis

This repository is not primarily about downloading repositories. It is about normalizing skill discovery, installation, and update behavior across a large and unstable matrix of agent products and source formats.

  • The center of gravity sits in src/add.ts, src/installer.ts, src/skills.ts, src/agents.ts, and src/cli.ts.
  • The defining constraint is ecosystem heterogeneity: GitHub, GitLab, well-known endpoints, local paths, plugin manifests, and node_modules are all supported.
  • This is not an agent runtime and not a general package manager; it projects skill folders into agent-readable filesystem layouts.

Core Modules

The heaviest modules are the ones that transform source repos into stable local skill directories.

Command Routersrc/cli.ts

Owns argv routing, help/banner UX, and most of the update/check workflow. It matters because update policy, legacy handling, and CLI re-entry all live here.

Install Orchestratorsrc/add.ts

Owns parsing, fetch/clone, skill selection, agent selection, install summaries, audit display, installation, and lock writes. This is the main product nexus.

Filesystem Installersrc/installer.ts

Turns discovered skill content into canonical and agent-specific directories with symlink or copy fallback semantics.

Skill Discoverysrc/skills.ts + src/plugin-manifest.ts

Finds valid skills in conventional directories, recursive layouts, and Claude plugin manifests while enforcing internal-skill and path-safety rules.

Agent Compatibility Matrixsrc/agents.ts

Encodes the supported-agent catalog, project/global directories, and install detection heuristics. This is a product asset, not just config.

Locking and Update Statesrc/skill-lock.ts + src/local-lock.ts

Persists global and project install state with different tradeoffs: richer update metadata globally and deterministic, merge-friendly project lock files locally.

Node Modules Syncsrc/sync.ts

Crawls node_modules for embedded skills and syncs them into project scope, showing the project wants skills to behave like distributable assets.

Distinctive Design Decisions

Canonical skill storage with agent projections

The installer writes to canonical .agents/skills locations and then symlinks or copies into agent-specific paths. That reduces duplication but forces the codebase to own symlink edge cases.

Universal-agent abstraction

Agents sharing .agents/skills are treated differently from agent-specific directories. This avoids duplicate installs and double listing, but it makes support logic path-taxonomy aware.

Two separate lock systems

Global installs track GitHub tree hashes and user-state metadata, while project installs optimize for deterministic, merge-friendly versioned JSON. That split is thoughtful and unusually product-aware.

Discovery by conventions before recursion

The code checks many well-known directories before doing full recursive search. This keeps common cases fast and predictable but expands the maintenance burden of the compatibility matrix.

Plugin manifest compatibility as first-class discovery

Claude plugin manifests are parsed directly instead of being treated as out-of-band metadata. That turns the CLI into an ecosystem bridge.

Update by CLI re-entry

The update flow shells back into the CLI's own add command. This preserves behavioral consistency with install, though it makes debugging more process-oriented.
